How sensor readings travel from hardware to your browser

The Bitstream data pipeline

Every sensor reading follows the same path:

Sensor
BMI270 etc. → Firmware
UART frames → Bridge
USB serial → Telemetry Provider
ws://127.0.0.1:9997 → Your UI
HTML / Studio

The event envelope

Every message from the provider uses the same JSON envelope:

{
  "type": "bitstream:sample"  // event type string
  "v":    1                  // protocol version — always 1
  "payload": { ... }      // event-specific data
}

Event types

EventDirectionWhen fired
bitstream:catalogprovider → youOn connect; contains all sensor field specs
bitstream:configprovider → youOn connect + after any config change
bitstream:sampleprovider → youEach decoded sensor measurement
bitstream:connectionprovider → youBoard connect/disconnect events
bitstream:staleprovider → youSensor has stopped publishing data
bitstream:helloprovider → youProvider ready + capabilities
bitstream:commandyou → providerR1: send a command to a sensor
bitstream:responseprovider → youR1: command acknowledgement

R0 vs R1 access tiers

R0 — Read

Subscribe to events. No special permissions needed. All demos in this workshop can run on R0.

R1 — Command

Send bitstream:command to change sensor config or mode. Ch 3 uses R1.

Transport: WebSocket vs postMessage

Your HTML page connects to the provider in one of two ways:

  • Standalone — new WebSocket("ws://127.0.0.1:9997"). Use when opening HTML directly in a browser.
  • Course Studio iframe — events arrive via window.addEventListener("message", …). Send bitstream:ready to trigger the handshake.

All demos in this workshop support both transports automatically.

The SHT40 sensor

The SHT40 is a precision temperature and humidity sensor with two fields:

Field keyUnitMinMaxNotes
temperatureC°C-40125Full sensor range
humidityPct%RH0100Bar fill = field value directly
💡
humidityPct is the only field where bar fill equals the value — because the physical range is already 0–100 %RH. All other sensors require range mapping.
Theory — Catalog-Driven Range Mapping
How to turn any sensor value into a 0–100% display percentage

Three range types — never mix them

TypeSourceUse for
Physical / catalogfields[].min, fields[].maxDefault bars and gauges
Gauge hintgaugeHints on sensor entryNarrower "typical motion" bands (e.g. gyro ±1 rad/s)
User display rangeUser specifies values + unitsOnly when user explicitly requests a custom scale

The range mapping formula

pct = clamp( (value − min) / (max − min) × 100, 0, 100 )
function clamp(n, lo, hi) {
  return Math.min(hi, Math.max(lo, n));
}

function valueToPercent(value, min, max) {
  if (max <= min) return 0;
  const t = (value - min) / (max - min);
  return clamp(t * 100, 0, 100);
}

// temperatureC = 25°C, range −40…125
// pct = (25 − (−40)) / (125 − (−40)) × 100 = 39.4%

Reading from bitstream:catalog at runtime

Always read min/max from the catalog event — don't hardcode values. Catalog arrives on connect:

let tempMin = -40, tempMax = 125; // safe defaults

function applyCatalog(payload) {
  const sht = payload.sensors.find(s => s.id === "sht40");
  const f   = sht?.fields.find(f => f.key === "temperatureC");
  if (f) { tempMin = f.min; tempMax = f.max; }
}

Mock fallback pattern

Every demo in this workshop starts a mock timer on load and stops it when the WebSocket connects successfully. This ensures the UI is always responsive:

let mockTimer = null;

function startMock() {
  if (mockTimer) return;
  mockTimer = setInterval(() => {
    onSample({ sensor: "sht40", fields: {
      temperatureC: 22 + Math.sin(Date.now() / 3000) * 5,
      humidityPct:  55 + Math.sin(Date.now() / 5000) * 20
    }});
  }, 200);
}

function stopMock() {
  if (mockTimer) { clearInterval(mockTimer); mockTimer = null; }
}

Stale detection

When a sensor stops publishing, the provider sends bitstream:stale. Always handle this to grey out the gauge and show a warning:

// bitstream:stale payload example:
{ sensor: "sht40", lastHostMs: 1718034123456, staleAfterMs: 1500 }

// Handle it:
case "bitstream:stale":
  if (msg.payload.sensor === "sht40") markStale();
  break;

Use Cases & Guidelines
Real Industry 4.0 applications for temperature and humidity monitoring

Applications

HVAC & Building Automation

Monitor temperature and humidity across zones. Trigger alerts when values leave comfort bands (18–26°C, 30–60%RH). The event-driven model means alerts are immediate — no polling delay.

Cold-Chain Logistics

Pharmaceutical and food transport requires temperature logging with timestamp. Use payload.hostMs for timestamping — it is synchronized to host system time.

Cleanroom Monitoring

Semiconductor fabs require <45%RH. Connect the stale event to an alarm — if the sensor stops reporting, treat that as a fault condition, not silence.

Guidelines

  • Always display the physical unit alongside the bar percentage — "39% (25.2°C)" is more useful than just "39%".
  • Use a color shift for out-of-range values (red/yellow) rather than just a number change.
  • Implement stale detection — a frozen gauge at 50% is dangerous if it is stale.
  • Never infer "good" or "bad" from raw value — that depends on application context, not sensor physics.

Theory — bitstream:config & R1 Commands
How sensor configuration works and how to change it

The bitstream:config payload

{
  "type": "bitstream:config", "v": 1,
  "payload": {
    "hostMs": 1718034123456,
    "source": "firmware", // "firmware" | "simulator" | "draft"
    "sensors": {
      "sht40": {
        "enabled": true,
        "publishMode": "hybrid",  // "periodic"|"on_change"|"hybrid"
        "mask": 3,              // bitmask of active channels
        "maskLabels": ["temperatureC", "humidityPct"],
        "samplingIntervalMs": 500,
        "publishIntervalMs": 0,
        "deltaX100": 50,
        "expectedRateHz": 2.0
      }
    }
  }
}

Publish modes

ModeBehaviourBest for
periodicPublish every samplingIntervalMsHigh-rate sensors, dashboards needing constant refresh
on_changePublish only when value changes by ≥ deltaX100 × 0.01Slow-drift sensors, battery-powered nodes
hybridPeriodic floor + on-change burstsBalanced — ensures heartbeat even when value is stable

R1 commands for configuration

// Change sampling rate of SHT40 to 1000ms, periodic mode:
sendCommand("sensor.cfg.set", {
  sensor: "sht40",
  publishMode: "periodic",
  samplingIntervalMs: 1000
}, "cfg-sht40-1");

// Enable BMI270 gyroscope (mask bit 2 = gyro channels):
sendCommand("bmi270.mode.set", { mode: "raw" }, "mode-1");
// then after response.ok:
sendCommand("sensor.cfg.set", {
  sensor: "bmi270", mask: 2,
  publishMode: "periodic", samplingIntervalMs: 50
}, "cfg-bmi-gyro-1");

Use Cases & Guidelines
Configuration strategies for real IoT deployments
Battery-Powered Edge Nodes

Use on_change mode with a generous deltaX100 threshold. A temperature sensor checking for >0.5°C change before publishing reduces UART traffic by 10×.

High-Rate Vibration Analysis

BMI270 in periodic mode at 50 ms (20 Hz) captures machinery vibration. Use mask bit 1 (accel only) to halve the data volume if gyro is not needed.

Hybrid Mode for Dashboards

Use hybrid for live dashboards — a periodic heartbeat (every 5 s) keeps the display alive even when value is stable, while on-change events provide responsive updates.

Guidelines

  • Use maskLabels (not mask) to check which fields are active — it is human-readable.
  • Wait for bitstream:response with ok: true before assuming a command succeeded.
  • After a successful command, a new bitstream:config event will arrive — do not cache old values.
  • Never modify sensor config from background timers — only in response to explicit user actions.
Summary — Chapter 3
Sensor configuration patterns
  • bitstream:config arrives on connect and after every config change — always handle it
  • Three publish modes: periodic, on_change, hybrid
  • R1 command flow: send command → wait for response → handle new config event
  • Always use maskLabels array to check active sensor channels
  • Match requestId in response to identify which command completed
Sensor Studio — Node Editor Basics
Chapter 4 · 75 min · Visual programming for sensor data pipelines

What is a node editor?

A node editor lets you build data pipelines visually — without writing code. Each node performs one operation. You connect output ports to input ports to route data through the pipeline.

ℹ
Sensor Studio is the node editor built into Bitstream Studio. Open it from the main toolbar. Each node in the canvas represents one step in your data pipeline.
Theory — Nodes, Ports & Data Flow
The core concepts of visual sensor data processing

Node categories

CategoryRoleExamples
Sensor nodesSource — emit live sensor field valuesBMI270, SHT40, DPS368, BMM350
Math nodesTransform — compute on incoming valuesScale, Add, Multiply, Abs, Sqrt, Moving Average
Logic nodesDecision — boolean operations and comparatorsThreshold, AND/OR, Latch, Timer
Visualization nodesSink — display valuesGauge, Numeric, Progress Bar, Alert, Chart

Data flow model

Sensor Studio uses an event-driven push model: when a new bitstream:sample arrives, all nodes connected downstream receive updated values immediately. There is no polling.

SHT40
Sensor node
──temperatureC──▶
Scale ×9/5+32
Math node
──▶
Gauge
Viz node

Port types

  • Numeric scalar — single float value (most common: temperature, pressure, one axis)
  • Numeric vector — array of floats (e.g. accel XYZ as one port)
  • Boolean — true/false (threshold comparator output, logic gates)

You can only connect ports of matching types. The editor highlights valid targets in green when dragging a connection.

Execution model

Nodes execute in topological order — upstream nodes always run before downstream nodes. Cycles are not allowed. The entire graph updates each time any source node receives new data.

Building Pipelines — Math, Logic & Multi-Sensor Flows
Chapter 5 · 75 min · Combine multiple sensors into processing pipelines
Chapter goal

Build two Sensor Studio pipelines: an acceleration magnitude alert (BMI270) and a 3-axis magnetic compass (BMM350). Learn to combine async sensor streams and apply math transforms.

Vector magnitude — the key math operation

|a| = √(x² + y² + z²)

Use this to reduce a 3-axis vector to a single scalar magnitude. Essential for vibration analysis, fall detection, and orientation-independent motion detection.

Multi-sensor synchronisation

Different sensors publish at different rates. BMI270 may publish at 20 Hz while SHT40 publishes at 2 Hz. When combining them in a flow, the downstream node fires whenever either upstream fires — using the last known value for the slower sensor.

💡
Design principle: Build separate flows for sensors with very different rates. Combine them only at the visualization layer (e.g. display panel) not in the processing graph.

Signal conditioning

  • Filtering — Moving average removes high-frequency noise. Window size of 5–20 samples is typical.
  • Deadband — The on_change publish mode + deltaX100 acts as a hardware deadband filter.
  • Threshold hysteresis — Use a Latch node to prevent rapid on/off toggling near the threshold.

Use Cases & Guidelines
Industry 4.0 pipeline applications
Vibration Fault Detection

Acceleration magnitude spike > 2g indicates bearing fault or imbalance. Use a 10-sample moving average to suppress transients, then a threshold node for alerting.

Fall Detection

Free-fall produces |a| ≈ 0; hard landing produces |a| > 30 m/s². A two-threshold latch (low → fall detected → high → impact) gives reliable fall events.

Magnetic Anomaly Detection (BMM350)

Magnetic field disturbances indicate ferromagnetic objects nearby — useful for vehicle detection, metal part presence sensing, and compass-based orientation.

Pipeline design principles

  • One concern per node — a "magnitude then threshold" pipeline should use separate nodes, not a combined custom node.
  • Test incrementally — verify each intermediate output before adding the next node.
  • For high-rate streams (BMI270 at 20 Hz), add a moving average before any threshold to reduce spurious triggers.

// From sensor-catalog.v1.json gaugeHints:

function applyCatalog(payload) {

}

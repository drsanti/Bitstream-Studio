# Development tracker

Single place to record **what shipped**, **what is next**, and **incoming requirements** so anyone can reopen the project and continue without losing context. Edit this file as part of normal work (small updates beat stale plans).

**Agent / team habit:** Cursor rules (`.cursor/rules/development-tracker.mdc`) tell the AI to **read** this file before substantial extension work and to **update** it when work ships, when users add requirements, or when logging suggestions—see that rule for details.

## Tracker vs changelog

| Artifact                                                              | Purpose                                                       | When to update                                                                                   |
| --------------------------------------------------------------------- | ------------------------------------------------------------- | ------------------------------------------------------------------------------------------------ |
| **This file (`DEVELOPMENT_TRACKER.md`)**                              | Rolling plan: backlog, inbox, parking lot, short “done” notes | Whenever priorities shift, work ships, or a new requirement arrives                              |
| **Version changelog** (e.g. project changelog next to `package.json`) | Release-facing history tied to **extension version**          | Only when cutting a release **and** the team chooses to maintain it (project convention applies) |

Do **not** duplicate long prose between the two: use the tracker for **ongoing state**; use the changelog for **shipping notes** per version when you publish.

## Agent handoff (new machine / Cursor)

**Start here when switching PCs:** repo root **`AGENT_HANDOFF.md`** (clone steps, v0.1.0 milestone, doc index, session log). Then **`HOW_TO_RUN.md`** and **`docs/TELEMETRY_MODE_LIFECYCLE.md`** (Bitstream vs Simulator exclusivity). Keep **`AGENT_HANDOFF.md`** updated when closing a substantial agent session.

## How to use

1. **Completed work** — When a feature or fix is merged and usable, add a bullet under **Recently completed** with **`YYYY-MM-DD`** prefix (ISO date). Remove or check off the matching item in _In progress_ / _Planned_.
2. **Active work** — Keep **In progress** to one or a few concrete items so focus stays clear.
3. **Backlog** — Put upcoming work in **Planned / next** (ordered roughly by priority). Move longer shots to **Future / ideas**.
4. **New requests** — Add a row to **Incoming requirements (inbox)** with **mandatory `YYYY-MM-DD`**, source, one-sentence requirement, and **Status** = `inbox`. After triage, set Status to `accepted` (and copy into _Planned_), `deferred`, or `rejected` (short note in _Parking lot_ if needed).
5. **Blocked / decisions** — Use **Parking lot** for items waiting on hardware, vendor API, design choice, or another team.

### Conventions

- **Dates**: Always **`YYYY-MM-DD`** (calendar date in the team’s timezone is fine; be consistent).
- **Checklists**: Use `- [ ]` / `- [x]` where helpful.
- **One line per item** unless nested bullets add clarity.
- **Optional**: `(owner)` or milestone suffix — only if the team uses them.
